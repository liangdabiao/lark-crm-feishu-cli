# lark-crm: 飞书轻型 CRM 系统 Skill

> 通过自然语言对话，直接管理飞书多维表格中的 CRM 系统。

## 是什么

`lark-crm` 是一个 基于飞书CLI 的 Agent Skill，它将你的「轻型 CRM 系统」多维表格封装成对话式管理工具。你不需要记住任何表名、字段名或命令参数，只需用自然语言描述你的意图，skill 就会自动：

1. **理解意图** — 判断你是要查客户、建商机、记跟进还是看报表
2. **翻译命令** — 自动映射到正确的多维表格和字段
3. **执行操作** — 调用 `lark-cli base` 命令完成操作
4. **格式化输出** — 以友好的表格或摘要展示结果

### 零配置设计

本 skill **不硬编码任何 base-token 或 table_id**。不同用户使用时，只需提供飞书多维表格 URL，skill 会自动：

1. 从 URL 解析 `base_token`（`/base/` 后的字符串）
2. 调用 `+table-list` 发现所有表及其 `table_id`
3. 按表名关键字匹配：客户管理、商机管理、合同管理、客户跟进记录、销售人员管理
4. 缓存到会话中，后续命令直接使用
5. 飞书crm模板地址：https://www.feishu.cn/base/bascndXmwM3qJOVZjUR6LS4uSng?table=tblVw8OKO1Yz2Lcl&view=vew4RfmKci
6. cli和skill安装办法： https://mp.weixin.qq.com/s/TGPCMtW92zTrt11ivm0pcA?scene=1&click_id=76

换人使用、换表格，只需提供一个新 URL 即可。

### 三层架构

| 层级 | 职责 | 文件 |
|------|------|------|
| **意图层** | 自然语言理解、关键词匹配、路由到对应 Shortcut | `SKILL.md` |
| **业务层** | CRM 表映射、字段校验、可写/只读判断、关联 ID 查找 | `SKILL.md` + `references/crm-commands.md` |
| **执行层** | lark-cli base 原子命令执行 | `lark-base` skill |

### 为什么这样设计

**问题**：用户每次操作 CRM 需要记忆 6 张表的 table-id、几十个字段名和类型、关联关系、哪些字段可写哪些只读，还需要手动拼接 JSON。换人或换表格时，还得逐个修改配置。

**解决**：skill 把这些知识全部固化下来，并支持自动发现配置。用户只需要说"帮我查一下北京互联网行业的客户"，Claude 自动完成：
- 配置初始化 → 首次使用自动发现 base-token 和表结构
- 意图识别 → 客户查询
- 筛选构建 → `+record-list` 拉取后本地过滤
- 结果展示 → 格式化表格

## CRM 系统结构

### 数据模型

```
销售人员管理（8人）
    │
    │ 负责跟进
    ▼
客户管理（20+客户）─── 客户跟进记录
    │                      │
    │ 产生                  │ 关联
    ▼                      ▼
商机管理（10条商机）── 合同管理（5份合同）
```

### 6 张数据表

| 表名 | 用途 | 核心字段 |
|------|------|---------|
| 客户管理 | 客户信息和联系人 | 客户名称、行业、规模、城市、对接人、客户所有人 |
| 商机管理 | 销售商机和阶段追踪 | 客户名称、跟进阶段、业务价值、预计交易日期、销售人员 |
| 客户跟进记录 | 跟进活动日志 | 客户名称、跟进内容、跟进形式、跟进人员、跟进时间 |
| 合同管理 | 合同和签约信息 | 合同编号、商机名称、客户名称、签约日期、合同金额 |
| 销售人员管理 | 销售团队和区域 | 销售姓名、销售区域、销售 Leader |
| 💡使用说明书 | 操作指引 | 流程阶段、功能详情、实现方法 |

### 表间关联

- **客户 → 商机**：一个客户可以有多个商机（一对多）
- **客户 → 跟进记录**：一个客户可以有多条跟进记录（一对多）
- **商机 → 合同**：一个商机可以对应一份合同（一对一）
- **销售人员 → 客户/商机**：销售负责的客户和商机（通过 lookup 自动关联）

### 自动化字段

| 字段 | 类型 | 说明 |
|------|------|------|
| 客户名称查重 | formula | 自动检测重复客户名 |
| 是否为公海客户 | formula | 自动判断无人认领的客户 |
| 机会名称 | formula | 自动拼接"客户名-阶段-描述" |
| 最近跟进时间 | lookup | 从跟进记录表自动关联 |
| 销售leader | lookup | 从销售人员管理表自动关联 |
| 销售区域 | lookup | 从销售人员管理表自动关联 |
| 签约人员 | lookup | 从销售人员表自动关联 |

## 使用方法

### 首次使用

提供你的 CRM 多维表格 URL：

```
"https://xxx.feishu.cn/base/XXXXXX?table=tblXXX&view=vewXXX"
```

系统会自动解析 base-token 并发现所有表结构，之后即可直接使用。

直接用自然语言描述你想查什么：

```

"互联网行业的客户有哪些"
```

## 权限要求

| 操作类型 | 所需权限 |
|---------|---------|
| 查询客户/商机/合同/跟进记录 | base 只读权限 |
| 新增/更新客户/商机/合同/跟进 | base 编辑权限 |
| 聚合分析（+data-query） | base 完全访问权限（FA） |

## 限制与注意事项

1. **首次使用需提供 URL** — 系统通过 URL 自动发现配置，无需手动编辑文件
2. **data-query 需要管理员权限** — 聚合分析需要调用者拥有 base 的完全访问权限（FA），非管理员用户只能使用记录查询和写入
3. **关联记录需先查 ID** — 新增商机时需要关联客户，必须先通过客户查询获取客户的 record_id
4. **只读字段不可写入** — formula、lookup、系统字段（创建时间等）由平台自动维护，手动写入会失败
5. **list 类命令串行执行** — lark-base 的 `+record-list`、`+field-list` 等命令不能并发调用
6. **单次查询上限 200 条** — `+record-list` 每次最多返回 200 条记录，需要更多时使用 offset 分页
7. **403 权限错误** — 遇到 HTTP 403 时，需让 base 管理员通过飞书界面分享表格并授予「可编辑」权限
8. **配置变更** — 若用户说「地址换了」或提供新 URL，系统自动重新发现表结构

### v2.0.0 — 零配置重构

- 移除硬编码的 base-token、URL、table_id
